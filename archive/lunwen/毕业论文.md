# 基于微信小程序的智能点餐系统AI客服的设计与实现

## 毕业论文

---

**摘要**

随着移动互联网的快速发展和人工智能技术的广泛应用，餐饮行业的数字化转型已成为必然趋势。传统的餐饮点餐系统通常只提供基本的菜品浏览和订单管理功能，缺乏智能化的客户服务能力。用户在使用过程中遇到问题时，往往需要等待人工客服响应，服务效率低下且成本高昂。为了解决这一问题，本文设计并实现了一套基于微信小程序的智能点餐系统AI客服系统。

本系统采用Spring Boot作为后端框架，结合微信小程序前端技术，构建了一套完整的餐饮服务解决方案。系统创新性地集成了基于ModelScope平台的AI智能客服功能，采用Qwen3-1.7B大语言模型实现自然语言交互；设计了餐饮知识库RAG（检索增强生成）系统，为AI客服提供专业领域知识支持；实现了多模型容错机制，有效解决了大模型API调用不稳定的问题。

本文详细阐述了系统的需求分析、架构设计、核心功能实现以及关键技术原理。系统实现了微信登录认证、菜单浏览选购、订单管理、AI智能问答等完整功能。经过测试验证，系统运行稳定，AI客服响应迅速，能够有效解答用户关于菜品、订单、配送等方面的咨询。

**关键词**：微信小程序；智能点餐；AI客服；RAG；大语言模型；ModelScope

## 第一章 绪论

### 1.1 研究背景与意义

随着人们生活节奏的加快和消费习惯的改变，餐饮外卖服务已成为现代生活中不可或缺的一部分。根据中国互联网络信息中心发布的统计数据，我国在线外卖用户规模已超过5亿，市场规模持续扩大。微信小程序作为腾讯生态系统的重要组成部分，凭借其无需下载、即用即走的便捷特性，已成为餐饮行业数字化转型的重要载体。众多餐饮企业纷纷开发自己的微信小程序点餐系统，以提升服务效率和用户体验。

然而，当前的餐饮点餐小程序普遍存在客户服务能力不足的问题。用户在浏览菜品、下单支付、订单配送等环节经常会产生各种疑问，例如菜品的具体口味、优惠活动的详细规则、订单的处理进度等。传统的人工客服模式面临响应速度慢、服务时间受限、人力成本高等诸多问题，难以满足用户即时获取信息的需求。

人工智能技术的快速发展为大模型赋能餐饮客服提供了可能。以GPT、LLaMA、Qwen为代表的大语言模型在自然语言理解和生成方面展现出强大的能力，能够理解用户的自然语言提问并生成流畅、连贯的回答。将大语言模型应用于餐饮客服场景，可以实现7×24小时不间断服务，大幅提升响应速度，同时降低人工客服成本。

基于上述背景，本研究设计并实现了一套基于微信小程序的智能点餐系统AI客服系统。该系统以Spring Boot为后端框架，微信小程序为前端载体，集成ModelScope平台的Qwen3-1.7B大语言模型，实现了智能化的客户服务能力。系统还创新性地设计了餐饮知识库RAG系统和多模型容错机制，进一步提升了AI客服的服务质量和系统可靠性。

### 1.2 国内外研究现状

#### 1.2.1 智能客服技术发展现状

智能客服技术的发展经历了从基于规则的对话系统到基于深度学习的大语言模型两个主要阶段。早期的客服系统主要采用关键词匹配和决策树等技术，通过预设的问题模板和答案库进行回复。这种方式虽然实现简单，但对话能力有限，无法处理复杂多变的用户问题。

近年来，随着深度学习技术的突破，序列到序列（Seq2Seq）模型、注意力机制（Attention Mechanism）以及Transformer架构的相继出现，使得机器对自然语言的理解和生成能力大幅提升。特别是2022年以来，以ChatGPT为代表的大语言模型展现出惊人的对话能力，能够理解上下文、生成连贯文本、处理复杂任务，为智能客服带来了革命性的变化。

在商业应用领域各大科技公司纷纷推出企业级智能客服解决方案。阿里云推出了基于通义千问大模型的智能客服系统，腾讯云提供了基于腾讯混元大模型的客服机器人，百度则依托文心一言大模型构建了企业知识库智能问答系统。这些解决方案已在电商、金融、政务等多个领域得到广泛应用。

#### 1.2.2 RAG技术研究现状

检索增强生成（Retrieval-Augmented Generation，RAG）是大语言模型应用的重要技术范式之一。RAG技术通过结合信息检索系统和文本生成模型的优点，有效解决了大语言模型在知识时效性、领域专业性以及 hallucination（幻觉）问题上的局限。

RAG的基本工作流程包括三个核心步骤：首先是文档加载与分块，将原始文档按照一定规则分割成较小的文本块；其次是向量嵌入与存储，将文本块转换为高维向量表示并存入向量数据库；最后是检索与生成，当用户提问时，系统根据问题检索出最相关的文本块，将其作为上下文提供给大语言模型生成回答。

在RAG技术的研究方面，Meta AI研究院提出了Dense Passage Retrieval（DPR）技术，显著提升了文档检索的准确率。LangChain、LlamaIndex等开源框架的兴起，降低了RAG技术的应用门槛，使得开发者可以快速构建基于私有知识库的智能应用。当前RAG技术已在企业知识管理、法律文档检索、医疗问答等专业领域得到广泛应用。

#### 1.2.3 微信小程序开发现状

微信小程序是腾讯于2017年推出的轻量级应用形态，运行于微信内部，无需单独下载安装即可使用。小程序采用WXML（类似HTML）、WXSS（类似CSS）和JavaScript的开发模式，提供了丰富的原生组件和API，支持支付、地图、摄像头等硬件能力。

在餐饮行业，微信小程序已成为主流的点餐渠道。美团、饿了么等外卖平台均提供小程序版本，同时众多连锁餐饮品牌如海底捞、瑞幸咖啡、西贝等也开发了自有小程序，提供预约点餐、外卖配送、会员管理等服务。小程序凭借其便捷的访问方式、成熟的支付生态和强大的社交传播能力，成为餐饮企业数字化转型的首选平台。

### 1.3 本文主要工作

本文的主要工作是设计并实现一套基于微信小程序的智能点餐系统AI客服系统，具体包括以下几个方面：

第一，系统需求分析与架构设计。通过对餐饮点餐业务的深入调研，明确系统的功能需求和非功能需求，设计合理的软件架构。系统采用前后端分离的架构设计，后端基于Spring Boot框架构建RESTful API服务，前端采用微信小程序实现用户交互界面。

第二，基于ModelScope的AI客服实现。深入研究ModelScope魔搭社区的大语言模型API接口，设计并实现AI客服服务模块。系统采用Qwen3-1.7B作为核心语言模型，通过精心设计的Prompt工程，引导模型扮演专业的餐饮客服角色，为用户提供准确、友好的咨询服务。

第三，餐饮知识库RAG系统设计。构建面向餐饮领域的专业知识库，整合菜品信息、优惠政策、配送规则等内容。采用Spring AI框架的向量存储机制，实现基于语义相似度的知识检索。当用户提问时，系统首先检索知识库获取相关内容，再结合大语言模型生成最终回答。

第四，多模型容错机制实现。针对大语言模型API可能出现的调用失败、响应超时等问题，设计并实现多模型容错机制。系统配置了多个备选模型，当主模型不可用时自动切换到备用模型，确保服务的连续性。

第五，系统测试与验证。对实现的系统进行功能测试和性能测试，验证各项功能的正确性和系统的运行效率。

## 第二章 相关技术理论

### 2.1 微信小程序技术

#### 2.1.1 小程序架构概述

微信小程序是一种运行在微信客户端内的轻量级应用程序，其架构设计遵循"宿主环境"理念。小程序的运行依赖于微信客户端提供的运行环境，包括WebView渲染层和Native原生层两部分。WebView负责渲染小程序页面的视图结构，Native层提供各种系统能力和原生接口。

小程序的开发采用类似Web前端的技术栈，主要包括WXML（WeiXin Markup Language）、WXSS（WeiXin Style Sheets）和JavaScript三个部分。WXML是一种类HTML的标记语言，用于描述小程序页面的结构；WXSS是一种类CSS的样式语言，用于控制页面元素的视觉效果；JavaScript则负责处理页面的逻辑交互和数据处理。

小程序还提供了丰富的API接口，涵盖网络请求、数据存储、位置服务、支付功能、设备能力等多个方面。开发者可以通过wx对象调用这些API，实现各种业务功能。小程序的生命周期包括onLoad（页面加载）、onShow（页面显示）、onReady（页面初次渲染完成）、onHide（页面隐藏）、onUnload（页面卸载）等关键回调函数。

#### 2.1.2 微信登录流程

微信小程序提供了完整的用户登录认证体系，通过wx.login()接口获取临时登录凭证code，然后向开发者服务器发起请求换取用户标识和会话密钥。整个登录流程遵循OAuth 2.0协议的思想，但进行了适配和简化。

具体的登录流程如下：首先，小程序调用wx.login()接口获取code；然后，小程序将code发送到开发者服务器；服务器使用code向微信服务器请求openid和session_key；服务器生成自定义登录态Token返回给小程序；小程序将Token存储在本地，后续请求携带Token进行身份验证。

这种登录方式既保证了用户信息的安全性，又为开发者提供了灵活的用户身份管理能力。本系统正是基于这一登录机制，实现了用户的注册、登录和会话管理功能。

### 2.2 Spring Boot微服务架构

#### 2.2.1 Spring Boot框架概述

Spring Boot是Spring生态系统中的明星项目，旨在简化Spring应用的创建和部署过程。Spring Boot通过"约定优于配置"的设计理念，提供了自动配置、内嵌服务器、健康检查等开箱即用的功能，使开发者可以快速搭建企业级应用。

Spring Boot采用Maven或Gradle作为构建工具，通过starter依赖自动管理项目所需的各类库。开发者只需在pom.xml中声明所需的starter，Spring Boot会自动完成相关依赖的下载和配置。例如，spring-boot-starter-web用于构建Web应用，spring-boot-starter-data-jpa用于数据持久化，spring-boot-starter-actuator用于应用监控。

Spring Boot的另一个核心特性是自动配置（Auto Configuration）。Spring Boot会根据classpath中的依赖和配置文件中的设置，自动配置Bean和应用上下文。例如，当classpath中存在H2数据库驱动时，Spring Boot会自动配置内存数据库；当前置依赖存在时，会自动配置JPA实体管理器。

#### 2.2.2 MVC架构与RESTful API

Spring Boot遵循经典的MVC（Model-View-Controller）架构模式，将应用程序分为模型（Model）、视图（View）和控制器（Controller）三层。Model层负责业务数据和业务逻辑；View层负责数据展示（在本系统中为JSON数据）；Controller层负责接收请求、调用Service层处理业务、返回响应。

REST（Representational State Transfer）是目前主流的Web服务架构风格。RESTful API遵循资源导向的设计原则，每个URL代表一个具体的资源，使用HTTP动词（GET、POST、PUT、DELETE）表示对资源的操作。本系统全部采用RESTful风格设计API接口，例如：

- GET /api/menu/items：获取菜单列表
- POST /api/orders：创建订单
- GET /api/ai-chat/history/{userId}：获取用户对话历史

#### 2.2.3 MyBatis持久层框架

MyBatis是一款优秀的持久层框架，它支持自定义SQL、存储过程以及高级映射。MyBatis通过XML或注解的方式配置SQL语句，将Java对象与数据库表进行映射，避免了繁琐的JDBC代码编写。

与JPA/Hibernate等ORM框架相比，MyBatis提供了更灵活的SQL控制能力。开发者可以编写高度优化的SQL语句，充分利用数据库的特性。同时，MyBatis的映射规则简单直观，便于理解和维护。

本系统采用MyBatis作为数据持久层框架，通过Mapper接口和XML映射文件定义数据库操作。MyBatis的动态SQL功能可以处理复杂的查询条件，ResultMap可以完成复杂的对象映射关系。

### 2.3 大语言模型与Prompt工程

#### 2.3.1 大语言模型原理

大语言模型（Large Language Model，LLM）是基于深度学习技术训练的自然语言处理模型，其核心是Transformer架构。Transformer由Google团队在2017年提出，采用自注意力（Self-Attention）机制来建模序列中各token之间的依赖关系。

大语言模型的训练过程分为预训练（Pre-training）和微调（Fine-tuning）两个阶段。预训练阶段，模型在大规模文本语料上学习语言的通用规律，掌握词汇、语法、语义等基础知识。预训练任务通常采用语言建模（Language Modeling），即根据前文预测下一个词的概率。微调阶段，模型在特定任务的数据集上进行进一步训练，使模型具备完成特定任务的能力。

当前主流的大语言模型包括OpenAI的GPT系列、Google的PaLM系列、Meta的LLaMA系列、阿里巴巴的Qwen系列等。这些模型在参数规模、训练数据、评测性能等方面各有特点。本系统选择使用阿里巴巴开源的Qwen3-1.7B模型，该模型在中文理解和生成方面表现出色，且提供了开放的API接口。

#### 2.3.2 Prompt工程概述

Prompt（提示词）是用户与大语言模型交互的接口，Prompt工程（Prompt Engineering）是指设计和优化提示词以获得更好模型输出的技术。良好的Prompt设计可以显著提升模型的回答质量，而不当的Prompt则可能导致模型输出不准确或不相关。

Prompt工程的核心原则包括：明确任务目标、提供充分上下文、给出具体示例、设定输出格式等。在实际应用中，常用的Prompt技巧包括：

（1）角色设定：明确告诉模型它应该扮演什么角色。例如："你是一名专业的餐饮客服"。

（2）few-shot示例：在Prompt中提供少量问答示例，帮助模型理解任务格式。

（3）思维链（Chain of Thought）：引导模型展示推理过程，尤其适用于复杂问题。

（4）结构化输出：指定输出的格式要求，如JSON、列表、分点说明等。

本系统通过精心设计的Prompt模板，将用户问题、对话历史、用户订单信息等内容组织成完整的输入，引导Qwen3-1.7B模型生成专业、准确的客服回复。

#### 2.3.3 ModelScope魔搭社区

ModelScope（魔搭社区）是阿里巴巴达摩院倾力打造的AI模型开源社区，旨在为开发者提供丰富的预训练模型和便捷的模型服务。ModelScope汇集了大量来自阿里云、学术界和开源社区的优质模型，涵盖自然语言处理、计算机视觉、语音识别等多个领域。

在自然语言处理领域，ModelScope提供了包括Qwen系列在内的众多大语言模型。开发者可以通过ModelScope的API接口调用这些模型，无需自行部署和运维模型服务。ModelScope API遵循OpenAI兼容的接口格式，便于开发者快速迁移现有的应用代码。

本系统通过ModelScope API调用Qwen3-1.7B模型，实现了AI智能客服功能。系统向API发送包含用户问题和上下文的Prompt，ModelScope返回模型生成的回答内容。

### 2.4 RAG检索增强生成技术

#### 2.4.1 RAG技术原理

检索增强生成（Retrieval-Augmented Generation，RAG）是一种将信息检索与大语言模型生成相结合的技术架构。RAG的核心思想是：当用户提问时，首先从知识库中检索出与问题相关的内容，然后将检索结果作为上下文提供给大语言模型，由模型根据检索内容生成最终回答。

RAG技术有效解决了大语言模型的几个固有局限。首先是知识时效性问题：大语言模型的训练数据通常有时效限制，无法回答最新信息；而RAG可以从实时更新的知识库中获取最新内容。其次是领域知识问题：通用大模型可能缺乏特定领域的专业知识，而RAG可以通过构建领域知识库来补充。再次是幻觉问题：大模型可能生成看似合理但实际错误的内容，而RAG可以基于检索到的真实内容进行约束。

RAG的工作流程主要包括以下步骤：

（1）文档加载：将各类文档（PDF、TXT、Word等）读取为文本内容。

（2）文本分块：将长文本分割成较小的片段（chunks），便于后续处理和检索。

（3）向量嵌入：使用嵌入模型将文本片段转换为向量表示。

（4）向量存储：将向量存入向量数据库，建立索引。

（5）问题检索：当用户提问时，将问题转换为向量，在向量数据库中检索最相似的文本块。

（6）答案生成：将检索到的内容作为上下文，结合问题发送给大语言模型生成回答。

#### 2.4.2 向量数据库与文本嵌入

向量数据库是RAG系统的核心组件，专门用于存储和检索高维向量数据。与传统的关系型数据库不同，向量数据库擅长处理"语义相似性"检索任务，即找出与查询向量最相似的文档向量。

常用的向量数据库包括Milvus、Pinecone、Weaviate、Chroma以及本系统使用的Spring AI SimpleVectorStore（内存向量存储）等。这些数据库通常支持近似最近邻（ANN）算法，能够在海量向量中快速找到最相似的候选结果。

文本嵌入（Text Embedding）是将文本转换为向量表示的过程。常用的嵌入模型包括OpenAI的text-embedding-ada-002、阿里巴巴的text-embedding-v2等。嵌入后的向量通常具有几百到几千个维度，能够较好地捕捉文本的语义信息。语义相近的文本，其向量在高维空间中的距离也较近，这正是向量检索的理论基础。

#### 2.4.3 文本分割策略

文本分割（Text Chunking）是RAG系统中的重要环节。合理的文本分割策略可以提升检索效果和生成质量。常见的文本分割策略包括：

（1）固定长度分割：按照预设的字符数或token数将文本切分成固定大小的块。这种方法简单直接，但可能破坏语义完整性。

（2）句子级分割：按照句子边界进行分割，可以较好地保持语义完整性，但检索粒度可能过细。

（3）递归分割：采用多级分割策略，先按大段落分割，再按句子分割，兼顾粒度和完整性。

（4）语义分割：使用机器学习模型识别文本的主题边界，实现更智能的分割。

本系统采用Spring AI框架的TokenTextSplitter进行文本分割，配置了500个token的块大小和100个token的重叠区域，确保每个文本块都有足够的上下文信息。

### 2.5 本章小结

本章介绍了本系统设计和实现所涉及的关键技术理论。首先阐述了微信小程序的技术架构和登录认证流程；然后介绍了Spring Boot微服务框架的原理和RESTful API设计方法；接着详细讲解了大语言模型的基本原理、Prompt工程技巧以及ModelScope魔搭社区；最后介绍了RAG检索增强生成技术的原理和实现方法。这些技术为本系统的开发奠定了坚实的理论基础。

#### 4.1.2 技术选型

根据系统需求和技术特点，本系统进行了合理的技术选型：

后端技术栈采用Spring Boot 3.2.0作为主框架，Spring AI 1.1.2作为AI能力框架，MyBatis 3.0.3作为持久层框架，MySQL 8.0作为关系型数据库。AI能力方面，采用ModelScope魔搭社区的Qwen3-1.7B作为核心语言模型，使用Spring AI SimpleVectorStore作为向量存储。

前端技术栈采用微信小程序原生开发框架，使用WXML、WXSS和JavaScript进行页面开发。

部署方面，系统采用Docker容器化部署，使用Docker Compose编排MySQL、Redis和应用程序容器。

### 4.2 核心模块设计

#### 4.2.1 用户模块设计

用户模块的核心实体是User（用户），主要属性包括：用户ID、微信OpenID、昵称、头像URL、角色（普通用户/管理员）、状态（正常/禁用）、创建时间、最后登录时间。

用户登录流程设计如下：用户在小程序端调用wx.login()获取code，携带code请求后端登录接口；后端使用code向微信服务器换取openid；根据openid查询或创建用户记录；生成Token令牌并返回给客户端；客户端存储Token，后续请求携带Token进行身份认证。

会话管理采用JWT Token机制，Token包含用户ID、过期时间等信息，设置72小时有效期。用户登出时，后端使Token失效。

#### 4.2.2 菜单模块设计

菜单模块的核心实体是MenuItem（菜品），主要属性包括：菜品ID、名称、描述、价格、分类、图片URL、是否在售、销量、创建时间。

系统预设的菜品分类包括：热菜、凉菜、主食、饮品、汤类、海鲜等。菜品支持按分类筛选、关键词搜索、销量排序等功能。

用户收藏功能通过UserFavorite（收藏）实体实现，记录用户ID和菜品ID的关联关系，支持添加收藏和取消收藏操作。

#### 4.2.3 订单模块设计

订单模块涉及三个核心实体：Order（订单）、OrderItem（订单明细）、UserAddress（收货地址）。

订单实体的主要属性包括：订单ID、订单编号（唯一）、用户ID、订单状态、总金额、联系人、联系电话、收货地址、备注、创建时间、更新时间。

订单状态流转设计为有限状态机，包含六种状态：

（1）pending（待付款）：订单刚创建，等待用户支付。

（2）confirmed（已确认）：用户已支付，订单已确认。

（3）preparing（准备中）：商家正在准备菜品。

（4）delivering（配送中）：菜品已送出，等待送达。

（5）completed（已完成）：订单已完成，交易结束。

（6）cancelled（已取消）：订单被用户取消或系统取消。

订单明细记录每个订单包含的菜品信息，包括菜品名称、单价、数量、小计金额等。

#### 4.2.4 AI客服模块设计（核心创新点）

AI客服模块是本系统的核心创新模块，其设计包含以下几个关键组件：

**（1）AI模型调用组件**

负责与ModelScope API通信，发送Prompt并接收模型生成的回答。系统配置了Qwen3-1.7B作为主模型，同时支持GLM-4.7-Flash等备选模型。模型调用组件封装了HTTP请求构建、响应解析、错误处理等逻辑。

**（2）Prompt构建组件**

负责将用户问题、对话历史、用户信息等内容组装成符合要求的Prompt。Prompt模板设计包含以下要素：角色设定（"你是一名专业的微信点餐小程序的智能客服助手"）、对话历史、用户信息（订单数、待付款数、配送中数等）、用户问题、输出格式要求。

**（3）对话历史管理组件**

负责存储和检索用户的对话历史。系统使用MySQL数据库存储ChatRecord（对话记录），每次问答都保存用户问题和AI回答。对话历史组件支持获取最近N条对话，用于构建上下文。

**（4）转人工判断组件**

负责判断当前问题是否需要转接人工客服。判断逻辑包括：用户明确要求转人工、连续多次无法理解用户意图、AI回答不确定等场景。

#### 4.2.5 知识库模块设计（核心创新点）

知识库模块为AI客服提供专业领域知识支持，采用RAG架构设计：

**（1）文档加载器**

支持加载多种格式的文档，包括PDF、TXT等。PDF文档使用PagePdfDocumentReader解析，文本文件使用TextReader解析。

**（2）文本分割器**

将长文档分割成较小的文本块，便于向量化和检索。系统使用TokenTextSplitter，配置500个token的块大小和100个token的重叠。

**（3）向量存储**

使用Spring AI的SimpleVectorStore作为向量存储，将文本块转换为向量并存储在内存中。向量存储支持语义相似度检索，返回与查询最相关的Top-K个结果。

**（4）知识检索**

当用户提问时，系统首先将问题转换为向量，在向量数据库中检索相似度最高的知识条目，将检索结果作为上下文提供给大语言模型。

**（5）预置知识**

系统预设了餐饮领域的基础知识，包括招牌菜品介绍、优惠活动规则、配送服务说明、营业时间等信息。系统启动时自动加载这些知识到向量库中。

#### 4.2.6 多模型容错机制设计（核心创新点）

为保证AI服务的稳定性，系统设计了多模型容错机制：

**（1）模型配置**

系统支持配置多个AI模型，配置格式为模型ID列表，如：["qwen/Qwen3-1.7B", "ZhipuAI/GLM-4.7-Flash", "MoonshotAI/Kimi-K2.5"]。

**（2）自动切换**

当某个模型调用失败（网络错误、API限流、服务不可用等），系统自动切换到下一个模型。每个模型都有一次调用机会，循环尝试直到成功或所有模型都失败。

**（3）失败降级**

当所有模型都调用失败时，系统返回友好的错误提示："抱歉，AI服务暂时不可用，请稍后重试。"并记录详细的错误日志供运维人员排查。

**（4）模型索引管理**

系统维护当前使用的模型索引，当某个模型连续失败时，可以动态调整模型优先级，确保系统的鲁棒性。

**（1）ModelScope API调用**

系统通过HTTP REST Client调用ModelScope API，发送包含Prompt的请求，获取模型生成的回答。核心实现代码如下：

```java
private String callModelScopeApi(String prompt) {
    // 配置模型列表
    if (models == null || models.isEmpty()) {
        models = List.of("qwen/Qwen3-1.7B", "ZhipuAI/GLM-4.7-Flash");
    }

    int maxRetries = models.size();

    // 遍历尝试每个模型
    for (int attempt = 0; attempt < maxRetries; attempt++) {
        String currentModel = models.get(currentModelIndex);

        try {
            // 构建请求体
            Map<String, Object> requestBody = new HashMap<>();
            requestBody.put("model", currentModel);

            List<Map<String, String>> messages = new ArrayList<>();
            Map<String, String> userMessage = new HashMap<>();
            userMessage.put("role", "user");
            userMessage.put("content", prompt);
            messages.add(userMessage);
            requestBody.put("messages", messages);
            requestBody.put("stream", false);

            // 发送HTTP请求
            String responseJson = restClient.post()
                .uri(baseUrl + "/chat/completions")
                .header(HttpHeaders.AUTHORIZATION, "Bearer " + apiKey)
                .header(HttpHeaders.CONTENT_TYPE, MediaType.APPLICATION_JSON_VALUE)
                .body(requestBody)
                .retrieve()
                .body(String.class);

            // 解析响应
            Map<String, Object> responseMap = objectMapper.readValue(responseJson, Map.class);
            List<Map<String, Object>> choices = (List<Map<String, Object>>) responseMap.get("choices");
            Map<String, Object> message = (List<Map<String, Object>>) choices.get(0).get("message");
            String content = (String) message.get("content");

            // 清理响应内容
            content = cleanAIResponse(content);
            log.info("模型 {} 调用成功", currentModel);
            return content;

        } catch (Exception e) {
            log.warn("模型 {} 调用失败: {}", currentModel, e.getMessage());
            // 切换到下一个模型
            currentModelIndex = (currentModelIndex + 1) % models.size();
        }
    }

    // 所有模型都失败
    log.error("所有模型都调用失败");
    return "抱歉，AI 服务暂时不可用，请稍后重试。";
}
```

多模型容错机制的核心优势包括：

- **API限流处理**：当某个模型触发API调用频率限制时，自动切换到其他模型继续服务。
- **网络异常恢复**：当网络请求超时或失败时，尝试其他模型。
- **服务不可用降级**：当某个模型服务暂时不可用时，保持服务连续性。
- **负载分散**：多个模型可以分散请求负载，降低单个模型的调用压力。

```java
@Service
public class OrderStatusService {

    // 定义状态转换规则
    private static final Map<OrderStatus, Set<OrderStatus>> ALLOWED_TRANSITIONS = Map.of(
        OrderStatus.PENDING, Set.of(OrderStatus.CONFIRMED, OrderStatus.CANCELLED),
        OrderStatus.CONFIRMED, Set.of(OrderStatus.PREPARING, OrderStatus.CANCELLED),
        OrderStatus.PREPARING, Set.of(OrderStatus.DELIVERING, OrderStatus.CANCELLED),
        OrderStatus.DELIVERING, Set.of(OrderStatus.COMPLETED),
        OrderStatus.COMPLETED, Set.of(),
        OrderStatus.CANCELLED, Set.of()
    );

    public void updateStatus(Order order, OrderStatus newStatus) {
        OrderStatus currentStatus = order.getStatus();

        // 验证状态转换是否合法
        Set<OrderStatus> allowed = ALLOWED_TRANSITIONS.get(currentStatus);
        if (!allowed.contains(newStatus)) {
            throw new BusinessException("订单状态不允许从 " + currentStatus + " 转换到 " + newStatus);
        }

        // 更新状态
        order.setStatus(newStatus);
        order.setUpdatedAt(LocalDateTime.now());
        orderRepository.save(order);
    }
}
```

#### 5.3.2 网络请求封装

小程序封装了统一的请求方法，支持Token自动注入：

```javascript
const request = (url, options = {}) => {
  const token = wx.getStorageSync('token');

  return new Promise((resolve, reject) => {
    wx.request({
      url: BASE_URL + url,
      header: {
        'Content-Type': 'application/json',
        'Authorization': token ? `Bearer ${token}` : ''
      },
      ...options,
      success: (res) => {
        if (res.data.success) {
          resolve(res.data.data);
        } else {
          reject(res.data.error);
        }
      },
      fail: reject
    });
  });
};
```

#### 5.3.3 AI客服页面实现

AI客服页面是小程序的核心交互页面，主要功能包括：

- 展示对话列表
- 输入问题并发送
- 接收AI回答并显示
- 支持语音输入（可选）

```javascript
// 发送消息
async function sendMessage() {
  const question = this.data.inputValue;
  if (!question.trim()) return;

  // 添加用户消息
  this.setData({
    messages: [...this.data.messages, {
      role: 'user',
      content: question
    }],
    inputValue: '',
    loading: true
  });

  try {
    // 调用AI客服接口
    const response = await api.aiChat.ask({
      userId: this.data.userId,
      question: question,
      sessionId: this.data.sessionId
    });

    // 添加AI回答
    this.setData({
      messages: [...this.data.messages, {
        role: 'assistant',
        content: response.answer
      }],
      loading: false
    });
  } catch (error) {
    wx.showToast({
      title: 'AI服务暂时不可用',
      icon: 'none'
    });
    this.setData({ loading: false });
  }
}
```

## 第七章 总结与展望

### 7.1 工作总结

本文设计并实现了一套基于微信小程序的智能点餐系统AI客服系统，完成了以下主要工作：

第一，需求分析与系统设计。通过对餐饮点餐业务的深入调研，明确了系统的功能需求和非功能需求。采用前后端分离的架构设计，后端基于Spring Boot框架，前端为微信小程序，整体架构清晰合理。

第二，核心技术实现。深入研究了Spring Boot微服务开发、MyBatis持久层、RESTful API设计等技术，构建了稳定、高效的后端服务。

第三，AI客服功能实现。创新性地集成了基于ModelScope平台的Qwen3-1.7B大语言模型，通过精心设计的Prompt工程，实现了专业的餐饮智能客服功能。AI客服能够理解用户自然语言提问，结合对话上下文提供个性化服务。

第四，知识库RAG系统实现。设计了面向餐饮领域的专业知识库，采用Spring AI框架的向量存储机制，实现了基于语义相似度的知识检索。系统预设了丰富的菜品信息、优惠政策、配送规则等知识内容，有效提升了AI回答的准确性。

第五，多模型容错机制实现。针对大语言模型API可能出现的调用失败问题，设计了多模型容错机制。系统配置了多个备用模型，当主模型不可用时自动切换，确保服务的连续性和稳定性。

第六，系统测试与验证。对系统进行了全面的功能测试和性能测试，验证了各项功能的正确性和系统的运行效率。

### 7.2 不足与展望

尽管本系统实现了预期的功能，但仍存在一些不足之处，值得在未来进行改进：

第一，知识库更新机制。目前知识库需要手动更新，未来可以设计自动化的知识更新机制，支持从外部数据源实时同步最新信息。

第二，对话管理优化。当前对话历史存储在MySQL数据库中，当用户量增大时可能影响性能。可以考虑使用Redis缓存对话历史，提升检索效率。

第三，多模态交互。目前AI客服仅支持文字交互，未来可以扩展支持图片、语音等多模态交互方式，提供更丰富的服务体验。

第四，用户意图识别。当前主要依赖大语言模型理解用户意图，准确率有待提升。未来可以引入专门的意图识别模型，提升问题分类的准确性。

第五，系统监控与运维。当前系统缺乏完善的监控和告警机制，未来可以集成Prometheus、Grafana等监控工具，实现系统运行状态的实时监控。

综上所述，本系统为餐饮行业的智能化服务提供了有价值的探索和实践。随着人工智能技术的不断发展，相信智能客服将在更多场景得到广泛应用，为用户提供更加便捷、高效的服务体验。
